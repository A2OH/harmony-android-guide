# liboh_skia_rtti_shim

A tiny shared library that provides Itanium C++ ABI typeinfo symbols
(`_ZTI*`, `_ZTS*`) for the Skia classes that Android `libhwui` references
at link time. It exists so the OH system library
`/system/lib/platformsdk/libskia_canvaskit.z.so` can stay **unmodified**
while `libhwui.so` still links cleanly.

## Why this exists

- OH's `libskia_canvaskit.z.so` is built with `-fno-rtti`. It exports no
  typeinfo for Skia classes.
- Android `libhwui.so` is built with `-frtti` and contains ~48 classes
  that inherit from Skia bases (`SkCanvas`, `SkDrawable`, `SkPixelRef`,
  etc.). The compiler emits typeinfo for those subclasses, which
  references the base typeinfo as an external undefined symbol such as
  `_ZTI10SkDrawable`.
- Previously the project rebuilt all of Skia with `-frtti` and replaced
  the OH system .so. That path affects every OH component that uses
  Skia. This shim is the low-risk alternative.

## Why it's safe

An exhaustive audit of `frameworks/libs/hwui` (2026-04-12) found:

- **0** `dynamic_cast<Sk*>` call sites
- **0** `typeid(Sk*)` call sites
- **48** hwui-defined subclasses of Skia types (the link-time symbol
  originators)

libhwui never performs a C++ RTTI query on any Skia object at runtime.
It uses Skia's built-in type dispatch (e.g., `SkMatrix::getType()`)
where type identity is needed. Therefore the shim only has to satisfy
the *link-time* typeinfo symbol requirement; the runtime semantics of
`dynamic_cast` / `typeid` on pure-Skia objects are never exercised and
do not constrain the design.

## How it works

`skia_rtti_shim.cpp` defines minimal polymorphic stubs whose mangled
names match the real Skia classes:

```cpp
class SkDrawable { public: virtual ~SkDrawable(); };
SkDrawable::~SkDrawable() {}
```

The compiler emits a full set of symbols for each stub (vtable,
destructors, typeinfo, typeinfo name). The linker version script
`skia_rtti_shim.ver` globalizes **only** `_ZTI*` and `_ZTS*`; everything
else stays LOCAL and cannot escape to collide with the real Skia
symbols.

`skia_class_list.inc` is the canonical class list and is regenerated by
`discover_skia_rtti_syms.sh` from the observed undefined `_ZTI*` refs
of a libhwui link against OH's original `libskia_canvaskit.z.so`.

## Building

Primary path is the direct clang script on ECS:

```
# On oh-build
bash ~/adapter/build/compile_skia_rtti_shim.sh
```

Output: `~/adapter/out/skia-rtti-shim/liboh_skia_rtti_shim.so`

Alternate GN path (for future integration into `bundle.json`):

```
cd ~/oh && ./build.sh --product-name rk3568 --ccache \
    --build-target liboh_skia_rtti_shim
```

## Discovery workflow

When the Skia header set or class hierarchy changes, regenerate the
class list:

```
# On oh-build
bash ~/adapter/build/discover_skia_rtti_syms.sh
```

The script:

1. Re-links `libhwui.so` against **OH's original** `libskia_canvaskit.z.so`
   (not the `-frtti` rebuilt one), with strict `--no-undefined`.
2. Captures every undefined `_ZTI*` / `_ZTS*` reference from ld.lld's
   error output.
3. Demangles via `c++filt`, emits `SKIA_RTTI_CLASS(...)` lines.
4. Rewrites `skia_class_list.inc`.
5. Re-builds the shim + re-links libhwui. Iterates to fixed point
   (typically 1–2 rounds).

The fixed-point class list is then committed.

## Verification

After build:

```bash
# L1 — every undefined _ZTI*/_ZTS* in libhwui is resolved by the shim
readelf -s --dyn-syms out/aosp_lib/libhwui.so \
    | awk '$7 == "UND" && $8 ~ /^_ZT[IS]/ {print $8}' \
    | sort -u > /tmp/needed.txt
nm -D --defined-only out/skia-rtti-shim/liboh_skia_rtti_shim.so \
    | awk '$3 ~ /^_ZT[IS]/ {print $3}' | sort -u > /tmp/provided.txt
comm -23 /tmp/needed.txt /tmp/provided.txt   # should be empty
```

L2–L4 device tests: see `doc/skia_rtti_shim_design.html`.
