## 0. Executive summary

Westlake hosts unmodified Android APKs by treating the Android framework as an embeddable runtime engine. After validating in Phase 0 that an Android app's view tree could be rendered through a Westlake pipeline, the engine entered a multi-month period of per-class shim work that did not scale — every new app surfaced ~5 new framework-service gaps requiring custom shim code. The diagnostic was that the engine substituted at the framework-class level, where the API surface fans out across hundreds of classes.

**V1 (the binder pivot, 2026-05-12)** corrected the cross-process boundary: substitute at the **Binder service boundary** rather than at framework classes. A Westlake-owned `libbinder.so` + `servicemanager` + AOSP-generated `IXxxService.Stub` subclasses handle every framework-to-service call uniformly, with AOSP's same-process `Stub.asInterface` optimization eliding marshaling for in-process Java-to-Java calls. AOSP framework.jar runs unmodified across `Stub.asInterface` / Parcel / BinderProxy.

**V2 (the substrate refinement, 2026-05-13)** corrected the in-process Java boundary: real AOSP framework code on `-Xbootclasspath` NPEs without `system_server`'s cold-init field graph (`Activity.attach`, `Application.attach`, `Resources.updateConfiguration`, etc.). Planting those fields one at a time is unbounded "additive shimming" — the very anti-pattern V1 § 6.2 forbade, dressed up in service-substrate clothing. V2 places a second substitution boundary at `Activity.attach` / `Application.attach` / `Window.<init>`: Westlake-owned classpath-shadowed `Application`, `Activity`, `Resources`, `Window`, `PhoneWindow`, `DecorView`, `WindowManagerImpl` replace framework's cold-init paths; everything below those classes (`View`, `ViewGroup`, widgets, `Handler`/`Looper`, the autogenerated `IXxxManager.Stub` proxies) runs unchanged from `framework.jar`.

**Current state (2026-05-14):**
- M1-M4 binder substrate complete: musl + bionic `libbinder.so`, `servicemanager`, 6 Java services (Activity/Power/Window/Display/Notification/InputMethod) plus `WestlakePackageManagerStub`.
- M5 audio daemon AUDIBLE — 440 Hz tone reproduced via the AAudio backend.
- M6 surface daemon at 60 Hz vsync with memfd `GraphicBuffer` transport and Android 15 wire compatibility.
- V2 substrate landed: classpath-shadowed `WestlakeActivity` / `WestlakeApplication` / thin Resources / Window+PhoneWindow / DecorView / WindowManagerImpl deployed via `framework_duplicates.txt`; zero new `Unsafe`/`setAccessible`.
- **M7-Step2 (noice) and M8-Step2 (McD) both reach `MainActivity.onCreate(Bundle)` user-code body** via production `Instrumentation.callActivityOnCreate`. CR59 closed today (2026-05-14) fixed `Application.mBase=null` so both apps now complete `onCreate` cleanly. Regression suite 14/14 PASS.

Phase 2 (M9-M13) targets OHOS. The OHOS standard-system kernel already enables binder (`CONFIG_ANDROID_BINDER_IPC=y`); the daemons swap their AAudio/DLST-pipe backends to OHOS AudioRenderer / XComponent via pre-scoffolded `#ifdef OHOS_TARGET` blocks. Phase 2 budget: ~13 person-days expected (`CR41_PHASE2_OHOS_ROADMAP.md`), gated on hardware acquisition. CR60 (2026-05-14) pivoted dalvikvm to 32-bit ARM on DAYU200 because the board's userspace is 32-bit only; CR61 (2026-05-14) locked in the binder strategy post-CR60 — ship our own libbinder + servicemanager via `/dev/vndbinder`, do not federate with OHOS `libipc`/`samgr`.

The architecture is closer in spirit to a slimmed Anbox than to the original "engine-with-shim-layer", but ~5× lighter (no zygote, no init, no full HAL, no full system image, no isolation namespaces). Estimated resident footprint: ~150 MB active vs. Anbox's ~700 MB-1 GB.

## 1. Background — how we got here

### 1.1 The original engine (pre-pivot)

The pre-pivot Westlake engine substitutes at the **framework class** level:
- ~2,300 Java source files / ~300K LOC of `android.*`, `androidx.*`, and `com.westlake.engine.*` shims.
- `MiniActivityManager` is what `getSystemService(ACTIVITY_SERVICE)` returns — a thin reimplementation of AOSP's `IActivityManager` proxy.
- `WestlakeLauncher` (originally ~23K LOC) orchestrates app load, field seeding, render loop, McD bootstrap.
- `WestlakeFragmentLifecycle` (3,087 LOC, deleted in C1) — renderer-time reflective fragment-lifecycle driver containing per-app constants (`MCD_SIMPLE_PRODUCT_HOLDER_ID`, `MCD_ORDER_PDP_FRAGMENT`).
- `DexLambdaScanner` (~600 LOC, deleted with C1) — dex bytecode parser used to discover Lazy<T> types per app.
- OHBridge `liboh_bridge.so` — ~15 JNI bridges from Java/native into OHOS surface/audio/input edges.
- ~50 ART patches (`PF-arch-001..051`) for dalvikvm bug fixes.

Working apps: Counter, Tip Calculator, TODO List, MockDonalds, McDonald's (real). All single-Activity, simple lifecycle, minimal multi-fragment navigation.

### 1.2 The pattern that broke (and why)

Westlake hits a wall on apps with multi-fragment navigation, deep DI, or non-trivial NavController usage. **noice** (`com.github.ashutoshgngwr.noice`) — multi-fragment NavController + BottomNavigation + Hilt — surfaced the pattern:

1. App crashes at framework-service NPE.
2. Fix the shim or seed the field.
3. App advances ~50 dex bytecodes.
4. Crash at the next framework-service NPE.
5. Loop.

Recent tasks #86-#97 in the task tracker all followed this pattern, each unblocking ~50 dex bytecodes. The diagnostic culminated in `Service.getServiceWithMetadata()` returning null because Android 16's `ServiceManager.rawGetService` wraps results in a metadata-carrying object that the shim didn't produce — yet another shim opportunity. The pattern would continue.

**Root cause:** the shim layer's substitution point is wrong. AOSP framework code calls `IActivityManager` on a service handle obtained from `ServiceManager.getService("activity")`; the shim replaces `IActivityManager` with a thin Java reimplementation. **Every method AOSP framework wants to call, we have to know about and implement.** The fan-out is hundreds of classes × dozens of methods each. Codex 2nd-opinion review (2026-05-12) independently confirmed:

> (a) "The boundary is right: framework/classes plus mini system services inside dalvikvm, with OHBridge only at platform edges."
> (b) "[WestlakeFragmentLifecycle] is off-architecture: renderer-triggered fragment instantiation… duplicates FragmentManager.addFragmentInternal."
> (e) "Per-app McD constants in FragmentManager/LayoutInflater are architectural debt; they will keep distorting deductions until removed."

### 1.3 The epistemic correction — subtractive validation

We were doing **additive shimming**: observe an NPE, add a shim, observe the next NPE, add another shim. The right method is **subtractive validation**: start from a fully-working baseline (real Android, where noice renders perfectly) and remove layers one at a time, observing which removal first breaks rendering. The layer whose removal first breaks rendering is load-bearing; everything above was unnecessary work. This document codifies subtraction as the *only* permitted way to discover what Westlake needs to implement.

## 2. V1 — the binder substitution insight

### 2.1 Why Binder is the right cut

Binder is **the** uniform service interface in Android. Every framework-to-system-service call goes through it. If Westlake owns the Binder layer:

- **AOSP framework.jar runs unmodified.** The framework looks up services by name through `ServiceManager`, gets back an `IBinder`, calls `IXxxService.Stub.asInterface(binder).someMethod(...)`. Whether the implementation is in-process, another process, or elsewhere, the framework doesn't care.
- **AOSP's same-process optimization works for free.** `Stub.asInterface(IBinder)` checks `queryLocalInterface()` and returns the local Java object directly when the service Binder lives in the caller's process. No Parcel marshaling, no IPC, just a method call. In-process Java service impls don't pay Binder overhead.
- **AOSP native libraries run unmodified.** `libaudioclient.so`, `libcamera2_jni.so`, `libgui.so` link against `libbinder.so` and talk `/dev/binder`. A real binder (kernel + userspace) means they work.
- **Service versioning resilience.** AIDL transaction codes are stable within an AOSP version — a framework.jar swap doesn't require chasing renamed framework classes.
- **Single substitution point** for ALL service work, both Java and native. Replaces the scatter of shimmed framework classes + thin manager classes + JNI bridges + renderer bypasses.

### 2.2 The four ingredients

1. **`libbinder.so`** — Westlake-owned, AOSP source recompiled against musl (and bionic for the Android phone validation). Handles `Binder`, `BBinder`, `BpBinder`, `IPCThreadState`, `Parcel`, `ProcessState`. Talks to `/dev/binder` (or `/dev/vndbinder`) for cross-process IPC; in the common single-process case AOSP elides the ioctl via `localBinder()`.
2. **`servicemanager`** — AOSP source recompiled against musl. The "phone book" daemon that the kernel hands `binder context 0` to. Apps and services use it to register / look up named Binder objects.
3. **Java service implementations** — Java classes extending the AOSP-generated `IXxxService.Stub` classes. Live in-process inside dalvikvm. Registered via `ServiceManager.addService("xxx", new WestlakeXxxService())` at engine boot. Same-process optimization makes Java-to-Java calls effectively direct method dispatch.
4. **Native daemons** — separate native processes that implement specific service contracts over real Binder. Required for AOSP native libraries that go through libbinder (audio, camera, media, surface). Each daemon translates its service contract to OHOS APIs internally.

The boundary between Westlake and OHOS lives **inside each native daemon** — typically the bottom ~500 lines of C++ per daemon, behind `#ifdef OHOS_TARGET`. Everything above that line is AOSP-shaped Binder service code, identical across platforms.

### 2.4 The Parcel kHeader send/receive asymmetry

Every kernel-binder transaction payload starts with a 4-byte `kHeader` magic that `Parcel::readInterfaceToken()` validates before dispatching. AOSP picks one of `'SYST'` / `'VNDR'` / `'RECO'` / `'UNKN'` depending on the `__ANDROID_*` macros set at compile time. Westlake's libbinder lands in AOSP's "non-Android" branch, where AOSP defaults to `'UNKN'` — rejected by both Android's `system_server` peers (expect `'SYST'`) and stock OHOS vendor binder endpoints (expect `'VNDR'`).

**Send side resolution:** `aosp-libbinder-port/patches/0003-parcel-kheader-syst.patch` makes the choice selectable at compile time via the Makefile's `KHEADER_FLAG`:
- `-DWESTLAKE_KHEADER_SYST=1` → `'SYST'` (default; Phase 1 sandbox interop)
- `-DWESTLAKE_KHEADER_VNDR=1` → `'VNDR'` (Phase 2 stock-OHOS-vendor interop)
- neither defined → `'UNKN'` (AOSP default, unmodified)

**Receive side resolution (CR11 follow-up, 2026-05-12):** patch 0003 controls only the *send* side. On 2026-05-12 the OnePlus 6's `vndservicemanager` began writing `'VNDR'` on replies; Westlake's libbinder (baked `'SYST'`) rejected every transaction with `Expecting header 0x53595354 but found 0x564e4452`. `aosp-libbinder-port/patches/0004-parcel-accept-any-kheader-on-recv.patch` widens the receive-side comparison to set-membership over `{'SYST', 'VNDR', 'RECO', 'UNKN'}`. The send side still writes `kHeader` so peers can identify us, but the four-byte magic is no longer a gate. This mirrors AOSP's own behavior across partitions — the partition split is enforced higher up via SELinux, not via this magic.

| Operation | Where | Behavior | Rationale |
|---|---|---|---|
| Send | `writeInterfaceToken` | Writes `kHeader` (today `'SYST'` via `WESTLAKE_KHEADER_SYST=1`) | Peer can identify us in logs; patch 0003. |
| Receive | `enforceInterface` | Accepts `'SYST' | 'VNDR' | 'RECO' | 'UNKN'` | Cross-partition / cross-vendor interop; patch 0004. |

## 3. V2 — the in-process substrate refinement

### 3.1 What V1 said and what we heard

V1 said *"AOSP framework.jar runs unmodified"*. The intended reading: `ServiceManager.getService()`, `Stub.asInterface()`, and the marshaling parts of framework.jar run unmodified, because Westlake's libbinder + Stub-subclass services satisfy them.

What actually happened, starting around M4-PRE6: real `framework.jar` was put on `-Xbootclasspath` alongside `aosp-shim.dex`, and `framework_duplicates.txt` (CR23-fix § 49) stripped the shim's `Context` / `Activity` / `ContextImpl` / `LoadedApk` / `Resources` / `AssetManager` / `Configuration` classes so framework.jar's copies won class resolution. The goal: let AOSP's own framework classes run through to Westlake's Binder services.

**The mistake:** AOSP framework classes are not pure transport. `Activity.attach(...)` does ~120 lines of state setup from `mApplication`, `mResources`, `mWindow`, `mWindowManager`, `mInstrumentation`, `mUiThread`, `mMainThread`, `mActivityInfo`, `mIdent`, etc. — fields that AOSP `ActivityThread.handleLaunchActivity` builds during process boot from `system_server`-supplied `IBinder` handles, `ActivityClientRecord`, `LoadedApk`, `CompatibilityInfoHolder`. Without that bootstrap, every field is JVM-default null, and `Activity.attach`'s first reference dereferences null.

### 3.2 The drift in tabular form

| Milestone | Symptom | "Fix" applied | Architectural cost |
|---|---|---|---|
| M4-PRE12 (2026-05-12) | `AssetManager.getResourceText` → NPE on `outValue` (private final `mValue`) | Reflectively plant `mValue = new TypedValue()` and `mOffsets = new long[2]` | +33 LOC; advances ~200 dex bytecodes |
| M4-PRE13 (2026-05-12) | `Locale.toLanguageTag()` NPE; synthetic `Configuration.mLocaleList` null | Reflectively plant `LocaleList(Locale.US)` via `Configuration.setLocales` + direct mLocaleList field | +58 LOC; advances ~100 dex bytecodes |

### 3.3 Why "plant the missing fields" is unbounded

AOSP's process bootstrap (`ActivityThread.handleBindApplication`, `LoadedApk.makeApplication`, `ContextImpl.createAppContext`, `Resources.updateConfiguration`, etc.) populates approximately:
- `ActivityThread`: ~40 instance fields
- `LoadedApk`: ~25 fields
- `ContextImpl`: ~30 fields (12-arg internal ctor)
- `Resources` / `ResourcesImpl`: `mAssets`, `mDisplayAdjustments`, `mCompatibilityInfo`, `mConfiguration`, `mMetrics`, `mAccessLock`, `mLocaleAdjustments`, …
- `Configuration`: `mLocaleList`, `locale`, `mLocale`, `screenLayout`, `densityDpi`, `screenWidthDp`, `screenHeightDp`, `smallestScreenWidthDp`, `uiMode`, `orientation`, …
- `AssetManager`: `mValue` (private final TypedValue), `mOffsets` (private final long[2]), native handle, theme attribute cache, …
- `DisplayAdjustments`: `mCompatInfo`, `mFixedRotationAdjustments` (API 30+)
- `Activity` during `attach`: `mMainThread`, `mApplication`, `mIntent`, `mReferrer`, `mComponent`, `mActivityInfo`, `mTitle`, `mParent`, `mEmbeddedID`, `mLastNonConfigurationInstances`, `mWindow` (real PhoneWindow), `mWindowManager` (real WindowManagerImpl), `mCurrentConfig`, …
- `PhoneWindow`: `mDecor`, `mLayoutInflater`, `mContext`, `mForcedWindowFlags`, `mWindowStyle` (TypedArray resolved from `R.styleable.Window`), …

That's well over 200 fields. M4-PRE12/13/14 covered three. There is no architectural reason to expect the next 200 to converge faster than the first three.

### 3.4 The corrected architectural fact

**Real AOSP `framework.jar` code is going to NPE on cold-boot state. Either we plant the state (drift) or we don't run that code.**

V2 picks: **don't run that code.** Specifically: don't run `Activity.attach`, `Activity.attachBaseContext` (the framework one), `Window.<init>`, `PhoneWindow.<init>`, `Resources.updateConfiguration`, or any framework path that touches the `ActivityThread`-built bootstrap graph.

Westlake **runs unchanged, in-process, from `framework.jar`:**
- `android.view.View` and subclasses (with the `mAttachInfo == null` caveat — see § 3.7)
- `android.view.ViewGroup`
- `android.widget.*` (TextView, Button, ImageView — pure-ish leaf widgets)
- `android.os.Handler`, `Looper`, `MessageQueue` (de-risked by M4-PRE4 MessageQueue JNI)
- `android.os.Bundle`, `android.os.Parcel` (data classes)
- `android.util.*` (Log, TypedValue, …)
- `android.graphics.*` data classes
- the Binder marshaling layer (`Stub` / `Stub.asInterface` / `Parcel.writeStrongBinder` / `BinderProxy`)
- autogenerated `IXxxManager.Stub` / `IXxxManager.Stub.Proxy` classes that Westlake services extend

Westlake **classpath-shadows (does not run from framework.jar):**
- `android.app.Application`
- `android.app.Activity` (and transitively, by `extends Activity` chain, FragmentActivity / AppCompatActivity / ComponentActivity continue to work because they only call public/protected `Activity` API)
- `android.content.res.Resources` (thin Westlake-owned surface reading `resources.arsc` directly)
- `android.view.Window`, `com.android.internal.policy.PhoneWindow`, `com.android.internal.policy.DecorView`, `android.view.WindowManagerImpl` (stubs; no chrome)

### 3.5 The classpath-shadow technique

For each shadowed class, Westlake:
1. Defines `shim/java/<fqn>.java` with the same fully-qualified name as the framework class.
2. Adds the fqn to `framework_duplicates.txt`, which strips the framework's copy from `aosp-shim.dex`.
3. Arranges classpath order so the shim wins class resolution at runtime.

The app's `MainActivity extends AppCompatActivity extends FragmentActivity extends ComponentActivity extends Activity` resolves `Activity` to Westlake's class. AppCompatActivity / FragmentActivity / ComponentActivity (in androidx, not framework.jar) continue to work because they only call public/protected `Activity` API. `instanceof android.app.Activity` checks succeed.

The substrate inherits AOSP's public/protected API surface — ~200 methods on Activity, ~80 on Resources, ~10 on Theme, ~30 on AssetManager — all documented and stable since API 14. **No `mLoadedApk`-style internals**, no cold-init field cascade, no per-app branches.

### 3.6 Where the cut lives

**Real `framework.jar` code runs up to the point where the app's Application/Activity constructor exits.** Westlake-owned classpath-shadowed versions take over from there. Real framework code resumes inside `View`, `ViewGroup`, widget classes, `Handler`/`Looper`, Binder marshaling, and the autogen `IXxxManager.Stub` proxies — all run cleanly in-process because they don't expect `system_server` bootstrap state.

**Architectural payoff:**
- App's `Application.onCreate` runs, including Hilt-injected fields, which fire real Binder transactions into M4 services. (V1 payoff preserved.)
- App's `Activity.onCreate` runs, including `setContentView(int)` which inflates real layouts via Westlake's `WestlakeLayoutInflater` (generic, not per-app).
- Real androidx Fragment lifecycle runs — once `getActivity()` returns a proper `WestlakeActivity` with `findViewById`, `getResources`, `getSystemService`, the entire androidx.fragment graph works without per-app code. The 3,087 LOC of `WestlakeFragmentLifecycle` we deleted is **not replaced** — it was solving a problem we no longer create.
- Real `View` / `RecyclerView` / `ConstraintLayout` measure/layout/draw runs.

Per-app shimming becomes structurally impossible: every app touches the same `WestlakeActivity` / `WestlakeApplication` / `WestlakeResources`, generic across all APKs.

### 3.7 Row 13 caveat: real View / ViewGroup from framework.jar

`android.view.View` has a `mAttachInfo` field that's populated by `ViewRootImpl` when the View is attached to a Window. Many View methods early-return when `mAttachInfo == null`. **This is in our favor** — it's the framework's own "I'm not attached yet" sentinel. Westlake's Activity / Window doesn't attach a `ViewRootImpl`; `mAttachInfo` stays null; dependent View code short-circuits cleanly.

The one wrinkle: `View.invalidate()` walks up the parent chain and eventually calls `ViewRootImpl.invalidateChild(this, dirty)`. If `mAttachInfo == null`, the upward walk stops. Westlake's render loop polls the View tree at frame cadence (M6 design) rather than reacting to `invalidate()`.

### 3.8 The macro-shim anti-drift contract

The V2 design is enforced by an explicit contract (`feedback_macro_shim_contract.md` in agent memory) which every Builder brief includes verbatim. Forbidden:
- `sun.misc.Unsafe.allocateInstance(...)` on framework.jar classes
- `Field.setAccessible(true)` + reflective set on framework.jar internal fields
- "Planting" state on framework's `ResourcesImpl`, `AssetManager`, `Configuration`, `Theme`, `ActivityThread`, `LoadedApk`, `ContextImpl`, etc.
- Per-app branches (`if (pkg.equals("com.mcdonalds.app")) ...`)
- `try { realFrameworkCold(...); } catch (NoSuchMethodError | LinkageError) { fallback() }` workarounds
- Adding new methods to `WestlakeContextImpl` (CR22 freeze)

Permitted: implementing public/protected API methods on Westlake-owned classes (the classpath-shadowed framework classes plus the Westlake-owned services), each method body one of (a) AOSP-default verbatim, (b) safe primitive (return null / false / 0 / empty list / no-op), or (c) delegation to another method on a class we own. `ServiceMethodMissing.fail(...)` is used for genuinely-unimplementable methods (CR2 pattern). A self-audit gate runs before completion.

`CR41_PHASE2_OHOS_ROADMAP.md` is the canonical pre-scoping document; `CR60_BITNESS_PIVOT_DECISION.md` and `CR61_BINDER_STRATEGY_POST_CR60.md` are the latest strategy decisions.

### 6.2 What swaps

1. **Kernel config:** verify binder on (likely already; one-line patch if not). → M9
2. **Build ABI:** rebuild every native artifact against `--target=aarch64-linux-ohos` (or `arm-linux-ohos` per CR60) instead of `--target=aarch64-linux-android24`. → M10
3. **Audio backend `.cpp`:** activate the already-scaffolded `OhosBackend.cpp` (M5 plan § 4.3) — ~250-350 LOC of `OH_AudioRenderer` adapter. → M11

**Approximately 80% of Phase 1's source code is reused unchanged**, and the remaining 20% is platform-bridge code pre-scoped behind `#ifdef`-selectable interfaces.

### 6.3 The 32-bit pivot (CR60, 2026-05-14)

On DAYU200, OHOS userspace is **32-bit only** — kernel runs aarch64 but every system binary is 32-bit ARM EABI5 against `/lib/ld-musl-arm.so.1`. The current 64-bit dalvikvm cannot `dlopen` any OHOS native library (XComponent, AudioRenderer, network) because they're all 32-bit, so it talks to them through the cross-arch IPC bridge (M6 daemon + AF_UNIX + memfd).

**Decision (CR60):** pivot dalvikvm to 32-bit ARM on DAYU200 because matching bitness eliminates the daemon for the production path. Reverse-pivot to 64-bit on a future 64-bit OHOS ROM is ~2-4 days of revalidation, not a rewrite. Total spike: 3-5 person-days, hard-bound.

**Discipline rules going forward** (to keep the pivot reversible):
- All native code uses `intptr_t` / `uintptr_t` / `size_t` for pointers. No `(int)pointer`, no `(long)pointer`.
- Build both arches in CI (cheap, catches regressions instantly).
- No `#ifdef __aarch64__` or `__arm__` branches in shim Java or JNI bridge unless absolutely necessary.
- `scripts/run-ohos-test.sh` auto-detects board bitness via `hdc shell getconf LONG_BIT`.

### 6.4 The binder strategy post-CR60 (CR61, 2026-05-14)

**We ship our own AOSP-derived `libbinder.so` + `servicemanager` for 32-bit OHOS, talking to the kernel binder driver via `/dev/vndbinder`. We do NOT integrate with OHOS's `libipc.dylib.so` / `samgr`.**

Empirical state of the DAYU200 board:
- Kernel binder driver live — all three nodes (`/dev/binder`, `/dev/hwbinder`, `/dev/vndbinder`) exist with world-rw DAC. Kernel side of M9 is already done by OHOS.
- OHOS does NOT ship `libbinder.so` — the AOSP userspace IPC ABI is absent. Apps that link against AOSP `IBinder`/`Parcel`/`Stub.asInterface` have no host library to call.
- OHOS ships `libipc.dylib.so` + `samgr` — a wire-incompatible system with different `IRemoteObject` semantics, different parcel format, different service registration model.
- `getenforce` → `Enforcing`. SELinux MAC is the gate, not Linux DAC permissions.

**Rationale for shipping our own libbinder:** Android APKs Westlake hosts (noice, McD) call AOSP binder APIs through framework.jar / aosp-shim.dex. Re-implementing every framework binder transaction against OHOS `IRemoteObject` would be a massive substitute-everything project — fundamentally the same problem the original macro-only decision was meant to avoid. Westlake's libbinder already passes 14/14 regression on Android phone; replacing a working component is unjustified. OHOS's `libipc` would not interoperate with `framework.jar`'s service stubs without per-service adapters — that's exactly the per-app-branch failure mode the macro-shim contract forbids.

**Coexistence model:**
```
DAYU200 process tree
├── samgr                          ← OHOS, on /dev/samgr_inner
│   └── serves: display_ability, audio_ability, …
├── westlake-servicemanager        ← OURS, on /dev/vndbinder
│   └── serves: activity, window, package, display, notification, input_method, power
└── dalvikvm (32-bit)
    └── talks to: westlake-servicemanager via our libbinder.so
    └── separately: dlopen libnative_window.so, libaudio_renderer.z.so for in-process OHOS APIs
```

The two service-manager universes don't see each other. AOSP framework code asking for `ActivityManagerService` reaches our implementation via our libbinder via `/dev/vndbinder`. OHOS code asking for `display_ability` reaches OHOS's implementation via libipc.

**M9 / M10 / M11 / M12 retargeting (per CR61 § 5):**

| Milestone | Original | Post-CR60 |
|---|---|---|
| M9 | Port `binder.ko` | **DONE by OHOS itself.** All 3 nodes present, kernel driver live. Verification only. |
| M10 | Cross-compile for `aarch64-linux-ohos-musl` | **Cross-compile for `arm-linux-ohos-musl`** (32-bit). Same Makefile, new `--target` triple. |

`setenforce 0` is **explicitly forbidden** as a "fix" — it disables MAC for the entire board and gives a false signal about production viability. SELinux policy work, if needed, escalates to documented per-domain rules.

## 7. Performance and tradeoffs

| Dimension | Pre-pivot (Option 2 incomplete) | Binder-pivot (Option 3 trimmed) | Anbox (Option 3 full) |
|---|---|---|---|
| Resident memory | ~50 MB | ~150 MB active | ~700 MB-1 GB |
| Cold boot | <1 s | ~2-3 s | ~20-30 s |
| Per-app launch | ~10 s (33 DEX) | ~3 s (warm VM + lazy daemons) | ~5 s (zygote fork) |
| AOSP native libs supported | ~0 (replaced via JNI shims) | ~all (via real binder) | ~all |
| WebView support | No | Yes (Chromium subprocess via real binder) | Yes |
| Multi-process app support | No | Yes | Yes |
| Kernel changes required | None | `binder.ko` (Phase 2; already enabled on OHOS) | `binder.ko` + namespace + cgroup + selinux |
| Per-app shim work | Linear with app count | Approaches zero (services converge) | Zero |
| Architecture proximity to real Android | Far | Close | Identical |

**Service-call latency:** in-process Java-to-Java via `Stub.asInterface` elision ≈ 50 ns (one virtual call). Java-to-daemon via real Binder ≈ 6 µs round-trip. Acceptable for service calls (typical app makes <1000/sec). Hot paths (SurfaceFlinger frame submit) use shared-memory buffer transport (memfd `GraphicBuffer`, BufferQueue) — same as AOSP.

**Frame budget:** 16.6 ms at 60 Hz. SurfaceFlinger compositor work happens in the daemon process; frame submit is one Binder transaction handing off a `GraphicBuffer` handle (kernel-duplicated fd, zero-copy data plane).

**Memory pressure:** lazy-launching daemons keeps idle footprint at ~75 MB (dalvikvm + servicemanager + Java services). Audio/surface daemons launch on first use. Camera/media daemons only launch for apps that need them.

## 8. Anti-patterns (codified)

These are enforced by every agent and every CR (`feedback_no_per_app_hacks.md`, `feedback_subtraction_not_addition.md`, `feedback_macro_shim_contract.md` in agent memory):

### 8.1 Per-app hardcoded shortcuts

No `if (className.equals("com.example.MyApp"))`, no `if (resourceId == 0x7f0b171c)`. Only generic Android API shims. Existing `MCD_*` constants (deleted in CR14/CR16) are the canonical violation example.

### 8.2 Additive shimming

Don't observe an NPE, add a shim, move on. The right reflex on an NPE: "what layer is missing? what would real Android provide here? where does that go in our architecture?" If the answer is "it goes in a service that lives behind Binder," add the service method (and verify it via Phase 1 sandbox); don't class-shim it.

### 8.3 Renderer-time bypasses

The renderer renders. It does not run lifecycles, seed fields, or inflate fragments. `WestlakeFragmentLifecycle.runLifecycleAndConvert` (deleted) is the canonical example of what NOT to do.

### 8.4 Reflection as the answer

Avoid `Unsafe.allocateInstance` for framework objects, avoid `Method.setAccessible(true)`, avoid `Field.get/set` on framework internals. These are signs of bypassing a framework path we should be participating in correctly. (Reflection on app-provided extension points, e.g., manifest-declared activity classes, is fine.)

### 8.5 Speculative completeness

Don't implement service methods we haven't observed being called. Discover via subtraction (the Phase 1 logging during noice run reveals exactly which Binder transactions hit each service). Implement only what's been observed.

### 8.6 No reflective field plants on framework cold-init objects (V2-added)

If you find yourself writing
```java
Field f = SomeFrameworkClass.class.getDeclaredField("mSomething");
f.setAccessible(true);
f.set(instance, reflectivelyConstructed());
```
on a framework class instance you just `Unsafe.allocateInstance`'d, **stop**. You are M4-PRE12/13/14 again. Substitute the class via `framework_duplicates.txt` instead.

### 8.7 No `try { realFrameworkCold(...); } catch (...) { fallback() }` patterns

If real framework code throws on cold-init, the right fix is to not call it. Wrapping in try/catch hides the architectural problem and accumulates dead branches. CR23-fix's `try { activity.attach(6 args) } catch (NoSuchMethodError | LinkageError)` was the canonical example — necessary at the moment, deleted in V2 Step 6.

### 8.8 No "almost framework" classes

`WestlakeContextImpl extends Context` is correct (implements an abstract surface). `WestlakeActivity extends Activity` via classpath shadow is correct (replaces a concrete class system-wide). But:
- Do NOT `class WestlakeActivityHelper { static void prepare(android.app.Activity a) { ... } }` that runs alongside framework Activity and partially substitutes its state. That's the deleted-with-WestlakeFragmentLifecycle pattern in different clothes.

### 8.9 No bypassing the V2 substrate via direct framework reflection

If V2 ships `WestlakeActivity`, then `WestlakeLauncher.mainImpl` MUST NOT reflectively construct framework's `android.app.Activity` and operate on it. The whole point of V2 is that there is exactly one `Activity` class at runtime — ours.

## 9. Open architectural questions

### 9.1 Does binder.ko apply to LiteOS-A targets?

Binder is a Linux kernel module. **OHOS LiteOS-A is not Linux** — it's a different kernel for small/embedded devices. binder.ko does not apply. If LiteOS-A targets enter scope, that's a separate architecture (likely "transpile to native ArkTS" — which has its own completeness problem). **Recommendation:** scope this pivot to standard-system OHOS (Linux-based) targets only.

### 9.2 BufferQueue ABI compatibility

AOSP's BufferQueue protocol (between client apps and SurfaceFlinger) changes occasionally between Android versions. Westlake's surface-daemon needs to honor the same ABI as the AOSP framework version shipped. Pinning to one framework version simplifies; supporting multiple needs version-specific transaction tables.

### 9.3 Permission model

Real Binder propagates the caller's UID/PID. Many framework permission checks call `Binder.getCallingUid()`. In our setup the caller is always our own UID, so checks pass trivially — but if an app checks "is this caller MY UID or SYSTEM_UID?", the answer matters. **Resolution:** boot-time configure caller UID as `SYSTEM_UID` (1000) for service-side calls, app UID for app-side calls. Use `IPCThreadState` to thread it through.

### 9.4 ashmem / memfd / shared-memory regions

AOSP services pass large data (audio buffers, surface frames, asset chunks) via shared-memory regions, not Parcel marshaling. Westlake targets **memfd**; a thin ashmem-shim maps ashmem APIs to memfd if a vendor library requires it. OHOS kernel ships both `CONFIG_MEMFD_CREATE=y` and `CONFIG_ASHMEM=y`.

### 9.5 ServiceManager handle 0 setup

The kernel-binder driver assigns `binder context 0` to whoever opens `/dev/binder` first with a specific ioctl. The boot orchestrator hard-orders servicemanager first, then waits for it to be ready before launching anything else.

### 9.6 Multi-Activity Intent dispatch

The app's `Activity.startActivity(Intent)` eventually calls `ActivityTaskManager.getService().startActivity(...)`. In V2, `WestlakeActivity.startActivity(Intent)` calls `IActivityManager.startActivity(...)` (M4a's Stub). M4a's implementation: queue the intent, mark it active, resolve the target activity class via local PackageManager, then trigger `WAT.launchActivityByComponent(component)`. **HIGH-rated risk** — new code path; will surface during M7-Step3 / M8-Step3 work.

### 9.7 AppCompatDelegate compatibility surface

`AppCompatDelegate.getDelegate(this)` reads `activity.getWindow()` to install its own DecorView. Westlake's `Window` is a stub. AppCompatDelegate's `installViewFactory(LayoutInflater)` calls `getLayoutInflater().setFactory2(this)`. AppCompatDelegate has ~15K LOC of compatibility shims; some paths read framework Activity fields. **MEDIUM-rated risk** — most likely place for the next drift temptation.

### 9.8 Hidden API access (`@hide` / `@SystemApi`)

Some app + library code reaches into `@hide` framework API. `WestlakeActivity` / `WestlakeApplication` should expose `@hide` methods the wild observes — discovered via running, not pre-implementing. CR2 `ServiceMethodMissing.fail` pattern handles this.

### 9.9 WebView

Chromium's WebView is fundamentally multi-process — renderer subprocess (sandboxed), GPU subprocess, network subprocess, IPC between them via real binder. This is Phase 3 work; the binder pivot is a prerequisite. Without real Binder, WebView is unreachable.

## 11. Glossary

- **Binder**: Android's primary IPC mechanism. A kernel module providing single-copy synchronous-and-async RPC between processes, with object references, death notifications, and UID propagation.
- **binderfs**: Linux feature (≥5.0) that lets userspace create independent binder contexts.
- **`/dev/binder`, `/dev/vndbinder`, `/dev/hwbinder`**: the character devices the binder module exposes for system / vendor / HIDL contexts.
- **`servicemanager`**: AOSP daemon that the kernel hands "context 0" to. The phone book for all named services.
- **`Stub.asInterface`**: AIDL-generated method that returns either the local Java implementation (same-process) or a marshaling proxy (cross-process). Critical optimization for the in-process service case.
- **`localBinder()`**: AOSP libbinder equivalent on the native side; same purpose.
- **`IPCThreadState`**: AOSP libbinder per-thread state holding the current transaction's caller info.
- **gralloc**: Android's graphics buffer allocator. Hardware HAL; Westlake bypasses it by having the surface-daemon allocate buffers via OHOS-equivalent mechanism (or memfd-based for portable testing).
- **AAudio / OpenSL ES**: Android audio APIs. AAudio is the modern low-latency one.
- **XComponent / OH_Drawing / OH AudioRenderer**: OHOS native APIs; targets of Westlake's daemon-internal bridge code.
- **classpath shadow**: declaring a class in `aosp-shim.dex` with the same fully-qualified name as a class in `framework.jar`, with classpath order arranged such that our class wins resolution. The framework copy of the class is unreachable.
- **cold-init field**: a field on a framework class whose value is set by `system_server`-driven bootstrap (`ActivityThread.handleBindApplication` / `handleLaunchActivity`), not by the class's constructor. Examples: `Activity.mApplication`, `ResourcesImpl.mDisplayAdjustments`, `Configuration.mLocaleList`.
- **plant**: reflectively setting a cold-init field on an `Unsafe.allocateInstance`'d framework object instance. The M4-PRE12-14 pattern. **Banned by V2.**
- **substrate**: the layer that satisfies framework public/protected API surface from below. Binder substrate (libbinder + servicemanager + services) satisfies `Stub.asInterface`'s expectations. Java substrate (WestlakeActivity + WestlakeApplication + WestlakeResources) satisfies the app's `super.foo()` expectations.
- **substitution boundary**: the line at which Westlake-owned code replaces AOSP code. V1 placed it at Binder (correct for cross-process). V2 places it ALSO at `Activity.attach` / `Application.attach` / `Resources` API surface inside dalvikvm (corrected for in-process).
- **drift**: the M4-PRE12-14-style pattern of advancing one cold-init field at a time, never converging. Identified and named in V2.
- **macro shim**: implementing public/protected API methods on classes we own (Westlake-shadowed framework classes + Westlake-owned services + Westlake-owned binder services). Each body is AOSP-default verbatim, safe primitive, or delegation. The only PERMITTED form of shim under V2.
- **micro shim**: any Unsafe.allocateInstance / setAccessible / framework-private-field-access / per-app-branch / catch-NoSuchMethodError-then-fallback pattern. **FORBIDDEN.**
