## 0. TL;DR

CR61.1 retracts the V2-OHOS prohibition on libipc / samgr (V3 reuses
HBC's `liboh_adapter_bridge.so` which links those innerAPI variants).
That retraction shapes a code disposition: some V2-OHOS-specific code
that was justified by CR61's ban is now obsoleted by V3 (archive). Some
remains useful as Android-phone V2 fallback (keep). Some sits between
the two and needs refactoring (rare).

**Final per-component count (subject to W13 validation):**

| Disposition | Components |
|---|---|
| **ARCHIVE** (move to `archive/v2-ohos-substrate/`) | 11 |
| **KEEP** (still valid for Android-phone V2 path) | 9 |
| **REFACTOR** (split / partial move) | 3 |
| **REVIEW WITH W11** (depends on W11 disposition) | 2 |
| **NO ACTION** (already correctly scoped) | 4 |

Total LOC delta: archive ~16K-18K LOC of Westlake-owned source + ~150 MB
of build outputs across `dalvik-port/build-ohos-*/`, `aosp-libbinder-
port/ohos-sysroot/`, and `aosp-shim-ohos.dex` build artifacts. No
deletions.

## 1. Inventory methodology

For each candidate component:

1. Trace its **justification origin** — was the component justified by
   (a) CR61 prohibition (own libbinder, no libipc), (b) V2-OHOS substrate
   ownership generally, or (c) something orthogonal (CR60 bitness,
   phone-only daemon work)?
2. Map to **V3-ARCHITECTURE.md §4** "what V3 deletes" row if applicable.
3. Cross-check **CR61.1 §5 migration impact table** for amendment-
   specific disposition.
4. Decide **archive / keep / refactor / no-action**.
5. Document risk if archive breaks Phase-1 Android path.
6. Write `git mv` command (do NOT execute).
