## TL;DR (5 bullets)

1. **HBC's CR61-equivalent is `technical_decision_overview.html` §2.10
   "innerAPI 优先于 NDK" (dated 2026-04-15)** — verbatim ruling:
   *"本项目与 OH 任何子系统对接时，一律链接 innerAPI / platformsdk /
   chipsetsdk 变体，不链 NDK 公共变体。"* Rationale: platform-level adapter
   = platformsdk consumer = whitelisted; NDK is the App-facing subset
   and "loses Android semantic fidelity" (HiLog `LogType`,
   `__android_log_set_minimum_priority`, etc.). Concrete audit found
   6 NDK-direct call sites + 7 NDK header refs + 2 NDK link-line args
   flagged for migration — Westlake should make the exact same call on
   `samgr` for the OHOS port.

2. **The L5 framework-patch pattern is 4 tiny diffs (228 lines total)**
   that delegate `ServiceManager.getService(...)` to
   `OHEnvironment.getXxxAdapter()` — and the whole concept of
   "non-BCP adapter classes loaded via system PathClassLoader" (so an
   adapter change costs **0 boot-image rebuild cycles**) is encoded as
   16 lines of reflective `newAdapterReflective(className)` in
   `OHEnvironment.java`. This is **the single highest-ROI borrowable
   pattern in the entire HBC project**.

3. **The reverse-bridge sandwich (C++ Stub → JNI → Java Bridge) has a
   load-bearing detail Westlake currently lacks**: HBC uses a
   `ScopedJniAttach` RAII inside every JNI helper to fix a class of
   crashes where OH binder threads attach-without-detach to ART
   (`thread exited without DetachCurrentThread` SIGABRT). Cited:
   `framework/activity/jni/ability_scheduler_adapter.cpp:35-95`,
   converted 33 call sites in one go.

4. **HBC's Skia compatibility play is a 12.6 KB symbol shim, not a
   24 MB Skia rebuild** — and the *audit methodology* is reusable for
   any "ABI gap, no semantic gap" problem. The shim defines 8 polymorphic
   `Sk*` classes with anchored virtual destructors, exports only `_ZTI*`/
   `_ZTS*` via a version script, and is regenerated by
   `build/discover_skia_rtti_syms.sh` (strict-link `libhwui` against the
   real OH `libskia_canvaskit.z.so`, grep undef `_ZTI*`, demangle, emit
   `SKIA_RTTI_CLASS(name)` lines). Iterative to a fixed point in 1-2
   rounds. Westlake hits the same shape of problem with `OH_Drawing_*`
   vs Skia and could copy the methodology cleanly.

5. **Westlake should NOT borrow (a) the whole AOSP-14 rebuild stack,
   (b) the 31-script in-place hwui-source `sed` chain — open admission
   it's `_deprecated/` but still load-bearing and unsedimented to
   .patch files — or (c) the legacy WMS path
   (`window_manager_use_sceneboard=false`). The first is 10× our budget,
   the second is the anti-pattern we already burned ourselves with on
   CR-W/X/Y/Z, and the third is a dead-end OH SKU.**

## The CR61-equivalent finding (load-bearing)

Westlake's open CR61 question is roughly *"do we depend on samgr +
libipc, or do we build something thinner?"* HBC answered exactly this
question 2026-04-15 in `doc/technical_decision_overview.html` §2.10
("OH 系统接口选型：innerAPI 优先于 NDK", reproduced verbatim above).

The five-bullet rationale (translated, condensed):

1. **"项目性质不是应用，是平台级适配层"** — *"The project is not an
   application, it is a platform-level adapter layer."* Adapter binaries
   live in `/system/`, so they are platformsdk consumers, not sandboxed
   apps. The "App-facing trimmed-down NDK" is the wrong target by
   definition.
2. **"OH 官方许可链接内部接口"** — OH's own `innerapi_tags =
   [chipsetsdk, platformsdk, sasdk]` is an *explicit whitelist*. Linking
   platform-tagged inner APIs is permitted, not a rule-bend.
3. **"NDK 裁剪会丢失语义保真度"** — NDK is a strict semantic subset.
   Concrete: NDK `OH_LOG_Print` only has `LOG_APP`; innerAPI `HiLogPrint`
   has 5 LogTypes that map cleanly to Android `log_id_t`'s 6 buffers.
   `__android_log_set_minimum_priority` has no NDK equivalent; innerAPI
   has `HiLogSetAppLogLevel`. *"Same logic applies to ipc_core, bms,
   ability_runtime — NDK only gives a minimum subset, insufficient for
   platform-level adaptation."*
4. **"二进制开销没有增加"** — innerAPI .so already exist on every device
   `/system/lib/` (system services use them); no new artifact ships.
5. **Fallback hierarchy** for the rare "innerAPI not yet up" cases:
   `innerAPI → libhilog_base.a static lib → kmsg/socket directly`.
   **Never NDK.**

The decision audits **6 current violations** (`framework/core/jni/android_log_hilog_bridge.cpp`
A1; `framework/hwui-shim/jni/atrace_compat.c` A2;
`framework/hwui-shim/jni/oh_choreographer_shim.c` A3;
`framework/hwui-shim/jni/oh_hardwarebuffer_shim.c` A4;
`framework/hwui-shim/jni/oh_native_window_shim.c` A5;
`framework/hwui-shim/jni/oh_minikin_shim.cpp` + `framework/surface/jni/oh_canvas_renderer.{h,cpp}` A6)
ranked by remediation cost. The forward-going rule: *"any new OH
sub-system integration must first audit innerkits/BUILD.gn for an
innerAPI variant; only fall back to NDK if no inner exists AND no
base static lib exists, and log the exception in §2.10."*

**Implication for Westlake CR61**: HBC's data point is that on real OH
(not Anbox-style chroot), `libsamgr_proxy.z.so` + `libipc_core.z.so`
are **mandatory, not optional**, for a platform-level adapter. Trying to
build a thin in-proc shim *inside an app* is a different bet (HAP-level
Candidate C); building a *V2-style substrate* on OH at all means
accepting samgr+libipc as link-time dependencies in `liboh_adapter_bridge.so`.
The verdict matches CR-DD's intuition but with a stronger justification
than we'd assembled.

## Pattern 1 — The 4 L5 framework patches (228 lines total)

### What HBC does

Four AOSP-14 `frameworks/base/core/java/android/` files patched, total
228 lines (`ls -l aosp_patches/.../*.patch | awk '{s+=$5}END{print s}'`).
Each one rewrites a `ServiceManager.getService(...)` singleton to call
`OHEnvironment.getXxxAdapter()` instead, with a fall-through to the
original on failure. Verbatim (`aosp_patches/frameworks/core/java/android/app/ActivityManager.java.patch:25-41`):

```java
private static final Singleton<IActivityManager> IActivityManagerSingleton =
        new Singleton<IActivityManager>() {
            @Override
            protected IActivityManager create() {
-               final IBinder b = ServiceManager.getService(Context.ACTIVITY_SERVICE);
-               final IActivityManager am = IActivityManager.Stub.asInterface(b);
-               return am;
+               // [OH_ADAPTER] Check if OH adaptation layer is available
+               if (OHEnvironment.isOHEnvironment()) {
+                   try {
+                       return new ActivityManagerAdapter();
+                   } catch (Exception e) {
+                       Log.w("ActivityManager",
+                           "OH adapter init failed, falling back to original", e);
+                   }
+               }
+               // Original AOSP implementation (fallback)
+               final IBinder b = ServiceManager.getService(Context.ACTIVITY_SERVICE);
+               final IActivityManager am = IActivityManager.Stub.asInterface(b);
+               return am;
            }
        };
```

The four patches:

| File | Singleton patched | Adapter target |
|---|---|---|
| `app/ActivityManager.java` | `IActivityManagerSingleton` | `ActivityManagerAdapter` |
| `app/ActivityTaskManager.java` | `IActivityTaskManagerSingleton` | `ActivityTaskManagerAdapter` |
| `app/ActivityThread.java` | `sPackageManager` (in `getPackageManager()`) | `PackageManagerAdapter` |
| `view/WindowManagerGlobal.java` | `sWindowManagerService` + `sWindowSession` (in `getWindowManagerService()` and `getWindowSession()`) | `WindowManagerAdapter` + `WindowSessionAdapter` |

Note: WindowManagerGlobal patches BOTH singletons in one diff (106 lines,
the biggest of the four), so this is **4 files / 5 singleton replacements**.

### Why it works

- All five fields are `Singleton<IXxx>` or static-cached, so the patch is
  a single-point intercept per service. No app code, no framework code
  elsewhere needs to know.
- Both adapter classes extend `IXxx.Stub` (class inheritance, not dynamic
  proxy — see Pattern 3), so the cast `Singleton<IActivityManager>` is
  satisfied at compile time.
- Fall-through preserves the AOSP-default path for non-OH builds. The
  same `framework.jar` is dual-target: stock AOSP device OR OH device,
  switched by `SystemProperties.get("persist.oh.adapter.enabled")`.

### How it adapts to Westlake

Westlake's stack is kitkat-dalvikvm + our own `WestlakeActivity`/
`WestlakeContextImpl`/`WestlakeApplication`, *not* AOSP framework.jar.
We never load `android.app.ActivityManager`'s real class definition;
we ship the substrate.

The L5 pattern is **the wrong shape** for our current path: we'd be
patching Java *source* to delegate, but our substrate's *replacement*
classes already exist. The Westlake-shape question is "where does
our `WestlakeActivityManager` call into?" — and the answer is
"into our own daemons" (M5/M6), not OH services.

**But if Westlake ever pivots to "run real KitKat framework byte-perfect
on top of OH"** (the HBC architectural shape, which the CR41 V2-on-OHOS
roadmap implicitly mirrors), this is the templated entry point: 4
KitKat singletons get the same L5 surgery, and our `OHEnvironment`-
equivalent gates them.

### Cost to borrow

- **As-is for current Westlake path**: 0 days, not applicable.
- **For hypothetical real-framework-on-OH path**: 1 day to mirror to
  KitKat 4.4 equivalents (KitKat has the same singletons; class names
  identical) + 1 day to author `WestlakeEnvironment.getXxxAdapter()`
  pattern matching `OHEnvironment.java:175-199`.

### Risk if skipped

For Westlake's current in-process substrate, **no risk** — we already
own all of these classes outright. The HBC L5 pattern only matters if
we want APK transparency *without* shipping our own substrate. That's
a strategic, not tactical, decision.

## Pattern 2 — OHEnvironment.java (the linchpin)

### What HBC does

`framework/core/java/OHEnvironment.java` (199 lines, complete file
pulled). Three structural pieces:

**Static loader (lines 17-20):**

```java
static {
    System.loadLibrary("oh_adapter_bridge");
}
```

This is the *only* native lib the L5 patches transitively trigger.
Once `OHEnvironment` is touched (by the patched `Singleton.create()`),
`liboh_adapter_bridge.so` is loaded into the ART process.

**Environment probe (lines 22-35):**

```java
public static boolean isOHEnvironment() {
    String prop = SystemProperties.get(PROP_OH_ADAPTER_ENABLED);
    if ("true".equals(prop))  return true;
    if ("false".equals(prop)) return false;
    return nativeIsOHEnvironment();
}
```

Dual-channel: system property + native probe. Lets the same
`framework.jar` decide at runtime whether to be in OH mode.

**Dual-classloader reflective factory (lines 130-173)**, with this
critical comment block that maps directly to the highest-velocity
engineering practice in the whole project:

```java
// L5 patches in AOSP framework.jar (ActivityManager / …) used to import +
// new-construct adapter classes directly, e.g.
//   import adapter.activity.ActivityManagerAdapter;
//   return new ActivityManagerAdapter();
// This required ALL adapter classes to live in BCP oh-adapter-framework.jar
// because framework.jar is BCP and Java imports resolve via the loading
// classloader. Result: every adapter change forced a boot-image rebuild
// (multi-day, risky).
//
// B.40 fix: only OHEnvironment stays in BCP. L5 patches now call the
// factories below, which use reflection + system classloader to
// instantiate adapter classes loaded from oh-adapter-runtime.jar (non-BCP).
// After this refactor, adapter changes only require recompiling
// oh-adapter-runtime.jar — no boot-image rebuild.
```

```java
private static Object newAdapterReflective(String className) {
    Throwable firstErr = null;
    // Path A — system classloader (PathClassLoader, non-BCP)
    try {
        ClassLoader cl = ClassLoader.getSystemClassLoader();
        if (cl != null && cl != OHEnvironment.class.getClassLoader()) {
            Class<?> cls = Class.forName(className, true, cl);
            return cls.getDeclaredConstructor().newInstance();
        }
    } catch (Throwable t) { firstErr = t; }
    // Path B — BCP classloader fallback
    try {
        Class<?> cls = Class.forName(className);
        return cls.getDeclaredConstructor().newInstance();
    } catch (Throwable t2) {
        System.err.println("[OHEnvironment] newAdapterReflective(" + className + ") both schemes failed");
        // ... error printing ...
        return null;
    }
}
```

Five public factories follow (lines 175-199): `getActivityManagerAdapter()`,
`getActivityTaskManagerAdapter()`, `getPackageManagerAdapter()`,
`getWindowManagerAdapter()`, `getWindowSessionAdapter()`.

### Why it works

- BCP holds 3 native method declarations + 3 API methods + 5 factory
  methods = **`OHEnvironment` is the only BCP-resident adapter class.**
- All real adapter logic lives in `oh-adapter-runtime.jar`, loaded by
  PathClassLoader (the same classloader that loads the APK). Adapter
  iteration = recompile one jar, push to device, restart appspawn-x. **0
  boot-image rebuild cycles.**
- Fall-through to BCP path (`Class.forName(name)`) means the old
  "everything in BCP" layout still works without code changes — pure
  build-time roll-back.

### How it adapts to Westlake

Direct port, with two changes:

1. **kitkat-dalvikvm has its own classloader hierarchy**
   (`BootClassLoader` → `PathClassLoader`); same shape, same trick works.
2. **Westlake doesn't currently have a "BCP vs non-BCP" split** because
   we ship our own classpath via `-Xbootclasspath` directly to dalvikvm.
   But we *could* add a Westlake "runtime.jar" loaded via `dalvikvm
   -cp` (non-BCP), and put 90% of our substrate iteration there. Right
   now any change in `WestlakeActivity.java` recompiles all of
   `framework_duplicates.txt` (12 KLOC). Splitting it lets iteration
   touch a 2-KLOC jar instead.

### Cost to borrow

- **2 days** to (a) author `WestlakeEnvironment.java` mirroring the
  reflective factory, (b) carve out the 5 most-changing classes
  (WestlakeContextImpl, lifecycle drivers) into a thin runtime jar,
  (c) update build/deploy to push only that jar on iteration.

### Risk if skipped

**Iteration-velocity tax.** Every substrate tweak forces full
framework_duplicates.txt rebuild + dalvikvm boot-cache invalidation. The
"why is testing slow?" complaint compounds across CRs. HBC quantified the
old-scheme penalty as "multi-day boot-image rebuild" and earned the right
to call this their highest-ROI iteration-velocity fix.

## Pattern 3 — Forward bridges (extends IXxx.Stub)

### What HBC does

Five Java classes, all sharing the same `extends <BCP_AIDL>.Stub`
inheritance pattern (`wc -l` confirmed):

| Bridge | LOC | Extends |
|---|---:|---|
| `framework/activity/java/ActivityManagerAdapter.java` | 2309 | `IActivityManager.Stub` |
| `framework/package-manager/java/PackageManagerAdapter.java` | 1670 | `IPackageManager.Stub` |
| `framework/window/java/WindowManagerAdapter.java` | 1298 | `IWindowManager.Stub` |
| `framework/window/java/WindowSessionAdapter.java` | 1115 | `IWindowSession.Stub` |
| `framework/activity/java/ActivityTaskManagerAdapter.java` | 1051 | `IActivityTaskManager.Stub` |
| **Total** | **7,443** | — |

Each adapter holds its own native handle + `static native` declarations
(no shared "AdaptationLayer" base) and tags methods `[BRIDGED]`,
`[PARTIAL]`, or `[STUB]`. Representative method
(`framework/activity/java/ActivityManagerAdapter.java:188-198`):

```java
// [BRIDGED] startActivity -> OH IAbilityManager.StartAbility(Want)
@Override
public int startActivity(IApplicationThread caller, String callingPackage, Intent intent,
        String resolvedType, IBinder resultTo, String resultWho, int requestCode,
        int flags, ProfilerInfo profilerInfo, Bundle options) throws RemoteException {
    logBridged("startActivity", "-> OH IAbilityManager.StartAbility");
    return bridgeStartAbility(intent);
}

private int bridgeStartAbility(Intent intent) {
    if (intent == null) return 0;
    IntentWantConverter.WantParams want = IntentWantConverter.intentToWant(intent);
    return nativeStartAbility(want.bundleName, want.abilityName,
            want.action, want.uri, want.extrasJson);
}
```

The four variants of `startActivity` (`Async`, `AsUser`,
`AsUserWithFeature`, `WithFeature`) all collapse to the same
`bridgeStartAbility(intent)` helper — 530+ method declarations, ~30
real bridge helpers.

### Why it works

- **Class inheritance** beats dynamic proxy: compile-time type
  checking, no `Method.invoke` reflection cost on every Binder call,
  and AOSP doesn't need to "trust" the proxy. The OS sees a normal
  `IActivityManager.Stub` subclass.
- **Per-adapter native handle**: each adapter owns its own `long
  mOhAbilityManager` initialized from `nativeGetOHAbilityManagerService()`.
  No shared singleton means no init-order coupling.
- **[STUB] tagging with safe defaults** lets the adapter ship 530+ method
  bodies without implementing them all on day 1. `getDefaultReturn(Class)`
  helper returns type-correct zero values (line 178-188).

### How it adapts to Westlake

We already do class inheritance (e.g., `WestlakeContextImpl`). HBC's
contribution is the **scale discipline** — 7,443 LOC of typed Stub
overrides is the price tag for full AIDL transparency. Westlake's V2
substrate is 12,403 LOC for a *different* surface (Activity/Window
in-process bind), but the broad lesson is the same: budget ~1-3 KLOC
*per AOSP service interface you transcode*.

For Westlake's per-AIDL tagging discipline (`[BRIDGED]`/`[PARTIAL]`/
`[STUB]` with `logBridged()`/`logStub()` helpers), **this is borrow-as-is**:
it's how you make a 2300-line file maintainable. We have a similar
pattern but inconsistent across files (some classes log, some don't).

### Cost to borrow

- **0 LOC change**, **1 day** to add `[BRIDGED]`/`[STUB]` comment tags +
  logging helpers to our existing substrate classes, with a CI grep
  ensuring every public method has one of the 3 tags.

### Risk if skipped

Maintenance friction. When McD probes a new method we haven't seen
before, having every method explicitly tagged tells you instantly
"this is bridged" (real call) vs "this is stub" (safe default) without
spelunking through 2 KLOC.

### What HBC does

6 reverse bridge sandwiches that translate **OH IPC callback → Android
callback object**. Files identified:

| C++ Stub adapter | Java Bridge | Routes to |
|---|---|---|
| `framework/activity/jni/app_scheduler_adapter.cpp` | `AppSchedulerBridge.java` | `IApplicationThread` (process lifecycle) |
| `framework/activity/jni/ability_scheduler_adapter.cpp` | `AbilitySchedulerBridge.java` | `IApplicationThread` (activity/service) + ContentProvider |
| `framework/activity/jni/ability_connection_adapter.cpp` | `AbilityConnectionBridge.java` | `IServiceConnection` (1:1) |
| `framework/window/jni/window_callback_adapter.cpp` | `WindowCallbackBridge.java` | `IWindow` (Android) |
| `framework/window/jni/session_stage_adapter.cpp` | `SessionStageBridge.java` | `IWindow` + `IApplicationThread` |
| `framework/window/jni/window_manager_agent_adapter.cpp` | `WindowManagerAgentBridge.java` | (no Android equivalent) |

The load-bearing detail is in `ability_scheduler_adapter.cpp:21-31` —
a 2026-05-06 refactor introducing `ScopedJniAttach` RAII that fixes
a class of crashes:

```cpp
// Original getJNIEnv() member function returned a JNIEnv* without a
// needsDetach signal, so OH binder threads passively attached by this
// adapter were never detached when the binder pool tore them down.
// ART CheckJNI fatal-aborted with
//   "thread exited without DetachCurrentThread"
// surfacing as cppcrash with thread name <pre-initialize> + SIGABRT
// shortly after activityResumed rc=0.
// The RAII class auto-detaches if (and only if) this scope did the
// attach. 33 call sites converted; getJNIEnv() member function removed.
```

The class body itself (lines 41-83 of the same file):

```cpp
class ScopedJniAttach {
public:
    explicit ScopedJniAttach(JavaVM* jvm) : jvm_(jvm), env_(nullptr), needsDetach_(false) {
        if (!jvm_) return;
        jint status = jvm_->GetEnv(reinterpret_cast<void**>(&env_), JNI_VERSION_1_6);
        if (status == JNI_OK) return;                   // Already attached
        if (status == JNI_EDETACHED) {
            JavaVMAttachArgs args = {JNI_VERSION_1_6,
                const_cast<char*>("OH_AbilitySchedAdapter"), nullptr};
            if (jvm_->AttachCurrentThread(&env_, &args) == JNI_OK) {
                needsDetach_ = true;
            } else { env_ = nullptr; }
        } else { env_ = nullptr; }
    }
    ~ScopedJniAttach() {
        if (needsDetach_ && jvm_) jvm_->DetachCurrentThread();
    }
    JNIEnv* env() const { return env_; }
    // ... non-copyable ...
};
```

The Java side (`AbilitySchedulerBridge.java:60-77`) has an equally
important detail — the **OH binder thread → Android main thread switch**:

```java
public void onScheduleAbilityTransaction(long tokenAddr, String wantJson,
                                          int targetState, boolean isNewWant) {
    String androidItem = mapLifecycleStateToItem(targetState);
    // ...
    IBinder activityToken = adapter.core.OhTokenRegistry.findByOhToken(tokenAddr);
    if (activityToken == null) {
        Log.w(TAG, "onScheduleAbilityTransaction: no Android token for OH tokenAddr=0x"
                + Long.toHexString(tokenAddr) + " — transaction dropped");
        return;
    }
    LifecycleAdapter lifecycle = LifecycleAdapter.getInstance();
    // ... map OH state to Android state ...
    lifecycle.dispatchOhLifecycleByActivityToken(activityToken, ohState);
}
```

Inside `LifecycleAdapter.dispatchOhLifecycleByActivityToken` (not pulled
but referenced by ability_scheduler_adapter.cpp), the dispatch posts a
`Handler(Looper.getMainLooper())` runnable. This is the thread-switch
that CR59 wrestled with in Westlake.

### Why it works

- **Two-layer (C++ Stub + Java Bridge) instead of one** means the C++
  side can satisfy OH's `IRemoteStub` abstract contract (vtable-based,
  C++-typed) while the Java side gets ergonomic typed object access.
- **`ScopedJniAttach` RAII** is the single most important
  cross-cutting detail. OH binder thread pool churns threads; ART
  CheckJNI hard-aborts on detach-without-attach. RAII guarantees
  symmetric attach/detach across all 33 call sites with zero per-call
  diligence.
- **Main-thread post in the Java bridge** lets the callback's *real*
  effect (start `ClientTransaction`, drive `Activity.onResume`) happen
  on the same thread that ran `Activity.onCreate`. Avoids the
  hwui-RenderThread vs binder-thread races that bit Westlake.

### How it adapts to Westlake

CR59 just fixed Westlake's version of the lifecycle-drive problem.
HBC's pattern is more disciplined:

1. **Adopt `ScopedJniAttach` everywhere we touch JNI from
   non-main threads.** Westlake's `Instrumentation.callActivityOnCreate`
   path runs on a worker; we currently rely on careful attach/detach
   discipline that's easy to break. ~80 LOC + audit pass converts our
   JNI helpers.
2. **Adopt the two-layer C++/Java sandwich** for any future inbound
   OH callback (e.g., when we add OH audio routing back to AppTrack).
   We're currently doing one-layer (Java callback object directly
   exposed via JNI fields) which works but loses the explicit
   main-thread post discipline.

### Cost to borrow

- **ScopedJniAttach RAII**: 1 day to author + audit our ~10 JNI helper
  call sites. **High-ROI safety.**
- **Two-layer sandwich pattern**: not needed today but ~3 days when
  we wire OH audio.

### Risk if skipped

JNI attach/detach asymmetry is a class-3 bug: hard to reproduce
(only fires when binder thread pool shrinks), hard to diagnose
(SIGABRT in `<pre-initialize>` thread), and corrupts JVM state. HBC
hit it after `activityResumed rc=0` — exactly where we are. *We will
hit this bug if we proceed without RAII.*

## Pattern 5 — AIDL transcode table (530+ methods)

### What HBC does

Each forward-bridge method body is a stub that calls one OH `inner_api`
method or returns a safe default. There is **no auto-generated transcode
table** — every method is hand-written and tagged. The discipline is
visible in `framework/activity/java/ActivityManagerAdapter.java` where
4 variants of `startActivity` collapse to one helper:

```java
@Override public int startActivity(...) {  return bridgeStartAbility(intent); }
@Override public int startActivityWithFeature(...) { return bridgeStartAbility(intent); }
@Override public int startActivityAsUser(...) { return bridgeStartAbility(intent); }
@Override public int startActivityAsUserWithFeature(...) { return bridgeStartAbility(intent); }

private int bridgeStartAbility(Intent intent) {
    IntentWantConverter.WantParams want = IntentWantConverter.intentToWant(intent);
    return nativeStartAbility(want.bundleName, want.abilityName,
            want.action, want.uri, want.extrasJson);
}
```

`IntentWantConverter` does the **structural transcode**: Android Intent
(component name + action + data URI + extras) → OH Want (bundleName +
abilityName + action + uri + extrasJson). The "extras" lossiness becomes
a hand-coded JSON serialization step.

The official interface mapping table from `CLAUDE.md` is the
authoritative many-to-many map:

| Android Interface | OH Target Interfaces |
|---|---|
| IActivityManager | IAbilityManager, IAppMgr, IMissionManager |
| IActivityTaskManager | IAbilityManager, IMissionManager, ISceneSessionManager |
| IWindowManager | IWindowManager(OH), ISceneSessionManager, DisplayManager |
| IWindowSession | ISession, ISessionStage, ISceneSessionManager |

| OH Callback Interface | Android Target |
|---|---|
| IAppScheduler | IApplicationThread (process lifecycle, memory) |
| IAbilityScheduler | IApplicationThread (activity/service) + ContentProvider |
| IAbilityConnection | IServiceConnection (1:1 mapping) |
| IWindow-OH | IWindow-Android (window state) |
| ISessionStage | IWindow-Android + IApplicationThread |
| IWindowManagerAgent | Internal (no Android equivalent) |

Note: many of the 530+ methods are **`[STUB]` with `logStub()`** — i.e.,
they explicitly return safe defaults because either (a) the OH side has
no equivalent, or (b) it's not on the HelloWorld critical path yet.
This is intentional engineering discipline, not "stub abuse" — the tag
makes the gap explicit.

### Why it works

- **Many-to-many mapping** is a *deliberate denial of bijection*:
  Android's IActivityManager is being decomposed into 3 OH services. The
  adapter is the routing layer. This avoids the temptation to "find one
  perfect OH service equivalent" — which doesn't exist.
- **Hand-coded extras JSON** is the only way to preserve Android Intent
  semantics across the wire. Reflection-based serialization would burn
  CPU per IPC.
- **`logStub()` discipline** turns a 530-method gap into a 530-line
  logcat audit trail. When something doesn't work, logcat tells you
  exactly which methods got hit but were stubbed.

### How it adapts to Westlake

Westlake's substrate currently fakes ~50 Activity API methods (we don't
have a 2309-line `WestlakeActivityManager` because we don't ship the
real `IActivityManager.Stub`). When CR41 Phase 2 OHOS pushes us toward
real-AIDL-on-OH territory, this 530-method discipline becomes the
template.

The **subset Westlake actually needs**: maybe 80-120 methods (Activity
lifecycle + Window create + Intent start + simple PM queries). HBC's
`bridgeStartAbility(Intent)` style decomposition cuts the workload
to ~25-30 real bridge helpers + bodies.

### Cost to borrow

- **As-is, today**: 0 days (we don't currently transcode AIDL).
- **If we pursue real-framework-on-OHOS (Phase 2 V2)**: budget
  3-4 person-weeks to author the 100-method Stub class for our subset,
  using HBC's tagging + helper-collapse style.

### Risk if skipped

**Without the tagging discipline**, a partial AIDL transcode is
indistinguishable from a fully working one until an app probes a
method we forgot. HBC's logcat-on-stub audit trail makes
silent-fail impossible.

## Pattern 6 — `liboh_skia_rtti_shim.so` (12.6 KB beats 24 MB Skia rebuild)

### What HBC does

Real source files (verified via `wc -l`):

- `framework/surface/jni/skia_rtti_shim/skia_rtti_shim.cpp` (~50 lines
  of macros)
- `framework/surface/jni/skia_rtti_shim/skia_rtti_shim.ver` (12 lines —
  the version script)
- `framework/surface/jni/skia_rtti_shim/skia_class_list.inc` (the
  generated class list — 8 polymorphic classes total)
- `build/discover_skia_rtti_syms.sh` (~200 lines — the audit-and-regen
  script)

The macro (`skia_rtti_shim.cpp:46-58`):

```cpp
#define SKIA_RTTI_CLASS(name)                          \
    class __attribute__((visibility("default"))) name  \
    {                                                  \
    public:                                            \
        virtual ~name();                               \
    };                                                 \
    name::~name() {}
```

The class list (`skia_class_list.inc:13-21`):

```cpp
SKIA_RTTI_CLASS(SkCanvas)
SKIA_RTTI_CLASS(SkNoDrawCanvas)
SKIA_RTTI_CLASS(SkPaintFilterCanvas)
SKIA_RTTI_CLASS(SkDrawable)
SKIA_RTTI_CLASS(SkPixelRef)
SKIA_RTTI_CLASS(SkWStream)
SKIA_RTTI_CLASS(SkExecutor)
// + nested SkBitmap::Allocator + SkBitmap::HeapAllocator
```

The version script (`skia_rtti_shim.ver:14-21`):

```
LIBOH_SKIA_RTTI_SHIM_1.0 {
  global:
    _ZTI*;
    _ZTS*;
  local:
    *;
};
```

The discovery script (`build/discover_skia_rtti_syms.sh:1-37`):
1. Link `libhwui.so` against the **real** OH `libskia_canvaskit.z.so`
   with `-Wl,--no-allow-shlib-undefined --no-unresolved-symbols`.
2. Grep link log for `undefined symbol: _ZTI…` / `_ZTS…`.
3. Demangle via `llvm-cxxfilt`.
4. Emit `SKIA_RTTI_CLASS(<name>)` lines, classify FLAT vs NESTED.
5. Rebuild shim, re-link `libhwui`. Iterate to fixed point (1-2 rounds).

### Why it works

- `libhwui` was built `-frtti` against Skia M116; OH ships
  `libskia_canvaskit.z.so` built `-fno-rtti` Skia M133. ABI gap = the
  link-time `_ZTI*`/`_ZTS*` typeinfo symbols.
- **Audit found 0 `dynamic_cast<Sk*>` and 0 `typeid(Sk*)` call sites in
  libhwui's 440 source files** — RTTI is link-time-only requirement,
  not runtime-behavior requirement. So fake typeinfos suffice.
- **Version script exports only `_ZTI*`/`_ZTS*`** — `_ZTV*` (vtables)
  and `_ZN*D[012]Ev` (destructors) are kept LOCAL. Critical: if those
  were GLOBAL, the dynamic linker could preempt real Skia calls into
  the shim's empty destructor bodies, causing crashes.
- **Visibility attribute** `__attribute__((visibility("default")))` is
  load-bearing — without it, clang under `-fvisibility=hidden` would
  mark `_ZTI/_ZTS/_ZTV` as hidden and they'd never make it to `.dynsym`.

### How it adapts to Westlake

Westlake doesn't (yet) cross-compile real `libhwui`. But the **audit
methodology** transfers to *any* "linker needs symbols, runtime won't
exercise them" gap:

- KitKat dalvikvm linking against OH-musl-built native deps: same
  shape. `discover_skia_rtti_syms.sh` adapted to dump `_ZTI*` from
  KitKat libdvm.so → emit symbol shim.
- Native-bridge IPC stubs: same shape. Build under strict-link, capture
  undefs, emit minimal `.cpp` with anchored destructors.

### Cost to borrow

- **Audit script methodology**: 1 day to author `westlake-undef-audit.sh`
  generalizing the strict-link + demangle + classify pipeline.
- **First concrete shim**: 0.5 day per ABI gap that arises.

### Risk if skipped

We'd "solve" each ABI gap one symbol at a time with hand-edited stub
files that drift. HBC's iterative-to-fixed-point script formalizes the
audit and re-runs it after every Skia/libhwui change. **The discipline
is more valuable than the specific shim.**

## Pattern 7 — ANativeWindow ↔ IBufferProducer 700-LOC shim

### What HBC does

`framework/window/jni/oh_anativewindow_shim.cpp` (708 LOC, verified).
Embeds AOSP `ANativeWindow` + `ANativeWindowBuffer` + `android_native_base_t`
**byte-exact** at the top of the file (lines 38-90), then provides
function-pointer wrappers (`aanw_dequeueBuffer`, `aanw_queueBuffer`,
`aanw_cancelBuffer`, `aanw_query`, `aanw_perform`, +
`_DEPRECATED` variants) that translate AOSP NDK semantics to OH
`OH_NativeWindow_*` calls.

Representative `queueBuffer` (lines 358-380):

```cpp
int aanw_queueBuffer(struct ANativeWindow* w,
                     struct ANativeWindowBuffer* buf, int fenceFd) {
    auto* a = as_adapter(w);
    if (!a) return -EINVAL;
    OHNativeWindowBuffer* ohBuf = unwrap_aosp_buffer(a, buf);
    if (!ohBuf) return -EINVAL;
    // ... (60 Hz probe logging trimmed for brevity) ...
    struct Region region = {0};  // empty region = full surface
    int32_t rc = OH_NativeWindow_NativeWindowFlushBuffer(a->oh, ohBuf, fenceFd, region);
    if (rc != 0) { ALOGW("queueBuffer rc=%d", rc); return -EIO; }
    return 0;
}
```

`query` (lines ~410-470) translates each `NATIVE_WINDOW_*` query code
to either an `OH_NativeWindow_NativeWindowHandleOpt(OH_OP_*)` call or
a safe constant default (e.g., `NATIVE_WINDOW_DATASPACE` returns
`ADATASPACE_SRGB = 142671872`).

`perform` (lines ~480-560) handles ~15 `NATIVE_WINDOW_*` operation
codes via `switch`/`va_arg` dispatch; unknown ops return `-ENOSYS` with
warn-log.

Plus a 396-LOC companion `framework/surface/jni/oh_graphic_buffer_producer.cpp`
that bridges Android `IGraphicBufferProducer` AIDL → OH
`IBufferProducer` for the actual buffer-queue side.

### Why it works

- **Byte-exact struct embed** (vs `#include`) lets the shim build under
  OH cross-compile without AOSP include paths, while still binding to
  the same offsets `libhwui.so` expects. The comment block at line 26-37
  is explicit: *"any drift from AOSP source layout would cause silent
  ABI mismatch — keep this block in lockstep with AOSP."*
- **Function-pointer wrappers** match AOSP's `ANativeWindow` vtable
  exactly; `libhwui` calls `window->queueBuffer(window, buf, fence)`
  with no recompile.
- **Safe-default `query`** returns plausible values for things OH
  doesn't expose (buffer age = 0 = "force redraw", min-undequeued = 1,
  etc.) — keeps the pipeline alive when probed.

### How it adapts to Westlake

Westlake currently uses SoftwareCanvas which is being deprecated for
DAYU200. The CR41 V2-on-OHOS roadmap (per memory `project_v2_ohos_direction.md`)
will need a **surface daemon on OHOS** — the producer side of that is
exactly this `OH_NativeWindow ↔ libhwui ANativeWindow` shim if we ever
host real AOSP libhwui (we won't, per V2 decision), or a much simpler
**Westlake-substrate ↔ OH IBufferProducer** shim if we author our own
SkiaCanvas (which we're doing per CR-DD).

The borrowable piece is the **structural pattern**:
1. Byte-exact embed the producer-side struct your substrate expects.
2. Function-pointer-wrap each method into the OH NDK call.
3. Safe-default `query` for anything not directly mapped.
4. Probe-log every Nth `queueBuffer` to confirm pipeline aliveness.

### Cost to borrow

- **For our own simpler producer shim**: 2 days to author a
  `westlake_buffer_producer.cpp` doing the same shape (DRM master client
  → OH IBufferProducer) without needing the ANativeWindow ABI surface.

### Risk if skipped

If our DRM-inproc-bridge path on DAYU200 hits the same "fence/age/
dataspace not exposed by OH NDK" gaps HBC hit, we'll spend a week
discovering them at runtime via crashes. HBC documented them in the
`query` switch statement — a free gap-list.

## Pattern 8 — 9-segment boot image

### What HBC does

`build/gen_boot_image.sh` (the source of truth, ~280 lines). Key
invariants encoded in error-banner comments at the top:

- **[B-1]**: `oh-adapter-framework.jar` has a *single source*
  (`out/adapter/`). Dual-source = boot-image vs jar-on-device checksum
  drift = SIGABRT.
- **[B-2]**: `art/runtime/asm_defines.h` must NOT be ARM32 version when
  host-running `dex2oat64`; script auto-removes if found.
- **[B-3]**: silent stale `.o` — patch + missing include = ninja skips
  re-link, .so MD5 unchanged but content stale. Verify with
  `strings .so | grep <marker>`.
- **[B-4]**: full-segment vs single-segment rebuild — full rebuild
  invalidates ALL `boot.oat` BCP checksums; OH services that boot off
  boot.oat then refuse to start. Use single-segment for daily iteration.
- **[B-5]**: libart + dex2oat + boot image are **strongly coupled**; any
  libart change requires dex2oat rebuild + boot image rebuild together.
- **[B-6]**: default `JARS` list must match `appspawn-x main.cpp:62`
  exactly. Mismatch = `ClassLinker::CheckSystemClass` abort:
  "Class mismatch for Ljava/lang/String;".

The canonical 9-segment order
(`framework/appspawn-x/src/main.cpp:111-122` for `kBootClasspath`, same
order in `gen_boot_image.sh:179`):

```
core-oj.jar
core-libart.jar
core-icu4j.jar
okhttp.jar
bouncycastle.jar
apache-xml.jar
adapter-mainline-stubs.jar  ← BEFORE framework.jar (key insight)
framework.jar
oh-adapter-framework.jar
```

The build invocation (`gen_boot_image.sh:~220`):

```bash
LD_LIBRARY_PATH=$HOST_OUT/lib64 \
LD_PRELOAD=$SIGCHAIN \
$DEX2OAT \
    --android-root=$HOST_OUT \
    --instruction-set=arm \
    $DEX_ARGS \                # --dex-file=<each jar's path>
    $LOC_ARGS \                # --dex-location=/system/android/framework/<jar>
    --oat-file=$OUTPUT/boot.oat \
    --image=$OUTPUT/boot.art \
    --base=0x70000000 \
    --runtime-arg -Xms64m \
    --runtime-arg -Xmx512m \
    --compiler-filter=speed
```

Output: 27 files (9 segments × `.art`/`.oat`/`.vdex`) at base address
0x70000000.

### Why it works

- **`adapter-mainline-stubs.jar` BEFORE `framework.jar`** in BCP: dex2oat
  prints a benign warning *"Skipping class X from framework.jar
  previously found in stubs.jar"* but runtime `Class.forName` of stub
  classes WORKS. Reversed order = silent runtime failure.
- **Base address 0x70000000** is the Android-Zygote convention; ART
  expects the boot image at that virtual address.
- **Fork-safe**: `appspawn-x` `fork()`s after boot image is mapped, so
  each child inherits the boot image via COW — zero per-child boot cost.

### How it adapts to Westlake

Westlake uses KitKat dalvikvm which has **dexopt** (boot.oat predecessor),
not `dex2oat` + boot image. The "9 segments" architecture doesn't
exist in KitKat. But:

- The **single-source-of-truth invariant for each jar** transfers
  directly (we already have multiple paths for framework_duplicates;
  formalizing single source would help).
- The **`strings .so | grep <marker>` verification** is universal —
  catches the same silent-stale-link bug in any cross-build pipeline.

### Cost to borrow

- **As-is, today**: 0 days (we don't have a boot image).
- **For potential Westlake AOT pre-load**: would be ~3 days to
  adapt the script shape if we ever pre-dex our substrate.

### Risk if skipped

For our current path, none. For any future "pre-build dalvikvm cache",
we'd want HBC's per-step error banners as a checklist.

## Pattern 9 — appspawn-x daemon

### What HBC does

`framework/appspawn-x/src/main.cpp` (311 LOC, full file pulled). Structure:

**Init service config** (`config/appspawn_x.cfg`, 60 lines):

```json
{
  "services": [{
    "name" : "appspawn-x",
    "path" : ["/system/bin/sh", "-c",
              "exec /system/bin/appspawn-x --socket-name AppSpawnX 2>>/data/local/tmp/parent_appspawnx.stderr"],
    "importance" : -20,
    "uid" : "root",
    "writepid" : ["/dev/memcg/perf_sensitive/cgroup.procs"],
    "socket" : [{ "name" : "AppSpawnX", "family" : "AF_LOCAL", "type" : "SOCK_STREAM",
                  "permissions" : "0660", "uid" : "root", "gid" : "appspawn" }],
    "ondemand" : true,        // ← critical
    "secon" : "u:r:appspawn:s0",
    "env" : [ /* ANDROID_ROOT, ANDROID_DATA, ANDROID_BOOT_IMAGE, ICU_DATA, ... */ ]
  }]
}
```

**Main entry** (`main.cpp:135-311`):

```
seedRequiredEnvs()  // setenv() workaround for OH init env[]>128 B bug
  → SpawnServer.initSecurity(sandboxConfig)
  → SpawnServer.createListenSocket()
  → AppSpawnXRuntime.startVm()          // dlopen libart.so + JNI_CreateJavaVM
  → AppSpawnXRuntime.preload()          // load ~30 framework classes
  → SpawnServer.setSpawnHandler(lambda):
        runtime.zygotePreFork()         // stop ART daemon threads
        pid = fork()
        if (pid == 0):
            ChildMain::run(msg, &runtime)   // child specialization + ActivityThread
        else:
            runtime.zygotePostForkCommon()  // restart ART daemons in parent
  → server.run()                        // blocks forever
```

**The critical `zygotePreFork`/`zygotePostForkCommon` mediation**
(`main.cpp:264-271`):

```cpp
// B.33 (2026-04-29): canonical Zygote fork mediation around fork().
// Without this, ART daemons (HeapTaskDaemon/Finalizer/Signal Catcher/
// Runtime worker x4) are alive in parent at fork time but vanish in
// child, leaving ART internal locks held by dead TIDs. Child main
// thread then blocks indefinitely in epoll_wait inside CallStaticVoidMethod
// (pre-Java thread attach), Java initChild never runs.
runtime.zygotePreFork();
pid_t pid = fork();
```

### Why it works

- **`ondemand: true`** + AF_LOCAL socket = appspawn-x doesn't consume
  RAM until first `aa start <android_bundle>` arrives. OH init listens
  for the socket; first connection wakes the service.
- **`setenv()` in main() instead of init cfg env[]**: OH init has a
  documented bug ("Z.1") where `env[].value` strings > 128 bytes
  overflow into the next env entry. Long values
  (BOOTCLASSPATH = ~400 B) are seeded by C code, not init JSON.
- **Zygote pre/post fork mediation** is the right pattern for "fork
  out of a process that has running JVM threads."

### How it adapts to Westlake

Direct precedent for "Android process spawned out of an OHOS init
service." Westlake's `dalvikvm` runs in an OH HAP process today (no
fork model). If we ever want to **pre-warm dalvikvm + framework
classes once + fork per app**, this is the template.

For DAYU200 Phase 2, a "westlake-x" daemon shape might be:
- ondemand listen on AF_LOCAL "/dev/unix/socket/WestlakeX"
- pre-load dalvikvm + thin substrate jars
- on launch request, fork() + setresuid + setcon + run app entry

That's literally the HBC main.cpp template with `libart.so → libdvm.so`.

### Cost to borrow

- **2-3 weeks** to author Westlake equivalent (this is the highest-cost
  pattern in the doc). Includes Zygote fork mediation, OH socket
  protocol design, SELinux label setup, sandbox config.

### Risk if skipped

For one-app-per-HAP, none. For multi-app coexistence + cold-start
latency targets (<500 ms), the fork-from-warm-VM model is the only
known path that hits those numbers. Skipping = each new app loads
KitKat framework from scratch (~2 s on rk3568).

## Pattern 10 — DEPLOY_SOP.md + anti-grep enforcement

### What HBC does

`deploy/DEPLOY_SOP.md` (12 KB, ~400 lines, verified pulled). Structure:

**Global 3 rules** (verbatim from top):

1. *"禁裸 PID / 禁碰 launcher"* — never `kill <pid>`; use
   `begetctl stop_service` or `killall <name>`. Never kill
   `com.ohos.launcher`.
2. *"推 /system 必走 staging"* — never `hdc file send <src> /system/...`.
   Always `send → /data/local/tmp/stage/<basename>`, `ls -la` to verify
   it's a `-rw-...` FILE (not a `drwx-` DIR — hdc bug), then `cp` into
   `/system`.
3. *"中止条件"* — hdc returns `connect-key` / timeout / `[Fail]Not a
   directory` / `ls` shows `drwx` → **stop immediately**, don't chain
   commands.

**Stage breakdown** (each stage has explicit success criteria):
- Stage 0: pre-checks (`/system/android` doesn't exist = factory state)
- Stage 1: backup 13 device originals as `.orig_${TS}` (no host-side
  backup needed — device-local copy is enough)
- Stage 2: stop 2 services (`foundation`, `render_service`) — NOT
  appspawn (kills hdc), NOT launcher (foundation stops it anyway)
- Stage 3a/3b/3c/3d/3e/3f: 6 categories of pushes, each with:
  - Source path → device path table
  - Symlink commands
  - Required `chcon` SELinux relabels with explicit "if you skip:"
    failure mode
- Stage 3.9: completeness + consistency MD5 verify
- Stage 3.5: reboot + service-up verification

**Anti-grep / verification** is embedded as `restorecon` after every
SELinux-sensitive push (Stages 3c, 3d, 3e, 3f). The SOP explicitly
notes (Stage 3f): *"OH toybox's `restorecon` does NOT support `-R`
('invalid args!'), must use `find -exec`"*.

The **`APK 透明适配原则`** ("APK transparency principle") in
`CLAUDE.md` lines 286-291 is HBC's anti-grep enforcement:

> 1. APK 源代码必须是纯原生 Android 代码。**绝不允许 `import adapter.*` 任何类**，
>    绝不允许调用 `OHEnvironment.*`、`ActivityManagerAdapter.*` 等适配层 API。
>    APK 不应感知"OH"或"适配层"的存在。
> ...
> 4. 验证适配层是否工作的方法:不是在 APK 里调用 `OHEnvironment.isOHEnvironment()`,
>    而是通过 logcat / dumpsys / hdc shell 等**外部观察手段**确认 native 调用走到了 OH 服务。

This is policy, not CI-enforced (no script found that actually greps
APK source for `adapter.*` imports — they rely on convention + code
review).

### Why it works

- **Staging cure for the hdc bug**: `hdc file send X /system/foo` where
  `/system/foo` doesn't exist creates a *directory* `foo` containing a
  file named the basename. Documented OH-specific bug. SOP-mandated
  staging always succeeds.
- **No-launcher-kill** preserves the foundation→launcher restart chain
  (launcher dies when foundation stops; reborns when foundation
  restarts). Manually killing launcher leaves it in a bad state.
- **SELinux relabel-after-cp**: file labels inherit from parent
  directory at `cp` time, not from `file_contexts`. Push `file_contexts`
  is necessary but NOT sufficient — must explicitly `restorecon` to
  walk the inode tree.
- **MD5 verify Stage 3.9** catches HCS-quirk silent truncations
  (`boot-framework.art` 23760896 bytes — exact value coded into Stage 0).

### How it adapts to Westlake

Direct, mostly. Westlake currently has no equivalent SOP — we have
`tools/qemu_boot.sh`, `prepare_images.sh`, scattered deploy scripts.
Authoring a similar staged-deploy doc would prevent the "I sent it but
it became a directory" class of bug we've hit on OnePlus 6.

For our DAYU200 deploys:
- Stages 0/1/2 transfer as-is.
- Stage 3 categories trim down (we have fewer .so files).
- SELinux dance applies to DAYU200 too.
- Stage 3.9 MD5 verify is universal.

### Cost to borrow

- **1 day** to author Westlake DEPLOY_SOP.md modeled on HBC's, with
  our specific file list + chcon labels.
- **0.5 day** to write a `verify-apk-transparency.sh` CI gate that
  greps Westlake's `app/` source for forbidden `westlake.*` imports.

### Risk if skipped

The hdc-creates-directory bug, the SELinux-label silent-failure, and
the truncated-MD5 bug have *all* burned us at least once. SOP formalizes
the lessons. Anti-grep enforcement is what keeps the "no per-app hacks"
rule alive across CRs.

## Pattern 11 — Bug catalog / lessons learned

### What HBC does

**Three layered documents** form the bug catalog:

1. **`doc/technical_decision_overview.html` §3** — "当前在用的临时手段
   / 捷径" (Current temporary measures / shortcuts). 15+ entries,
   each with: 手段 (measure) / 位置 (location) / 为什么不得不用 (why
   forced) / 退出条件 (exit condition) / 状态 (state). Includes both
   resolved items (struck-through) and active debt.

2. **`doc/compile_report.html` Appendix Z** — OH bug numbers
   (Z.1, Z.4.2, etc.) — each cross-referenced from code comments. E.g.,
   `init_service_manager.c:1023` env overflow = Z.1.

3. **`build/gen_boot_image.sh` top-of-file banner** (lines 1-90) —
   the **"B-1 through B-6 invariants"** (Pattern 8 above). Each invariant
   has: 现象 (symptom) / 根因 (root cause) / 措施 (countermeasure) /
   血训 (lesson date + 1-2 sentence narrative).

The discipline is encoded in `CLAUDE.md`:

> ## Root Cause Analysis Discipline (项目级铁律)
>
> Android 和 OH 都是成熟系统，遇到运行/链接/解析等问题时，**禁止从
> "质疑 Android 或 OH 现有实现的正确性"开始。** 第一嫌疑必须放在
> adapter 层。具体排查顺序：
>
> 1. **适配层的转换哪里没有处理好** — adapter 自己写的 Java/Native
>    代码、JNI 桥接、stub/shim、IPC 接口转换、Java↔OH IPC 数据结构映射。
> 2. **编译参数与 Android/OH 原生不一致** — cross-compile 时缺的 `-D`
>    宏、wrong target triple、缺的 include path、`__ANDROID__` / `__OHOS__`
>    被错误条件分支选中、ABI flag 不匹配。
>
> 只有上述两个方向都排查过且证据明确指向上游，才允许提出"原生实现可能
> 有 bug"的假设——而且必须先做 control 实验（同一调用从两个不同
> adapter 路径触发对比）才能成立。
>
> 历史血训（2026-04-29 B.21 r3）：连续两次在 framework-res 解码链路上
> 把根因错指到 libandroidfw 和 OH libskia_canvaskit，实际真因都是
> adapter 层。

### Why it works

- **§3 shortcuts table** is the project's debt ledger. Every workaround
  is visible; nothing hides.
- **B-1..B-6 invariants at point-of-use** (in `gen_boot_image.sh`)
  means the dev who edits the file SEES the rules; they don't live in
  a separate wiki that nobody reads.
- **"Blame adapter first" RCA discipline** with explicit "历史血训"

## Pattern 12 — Build pipeline

### What HBC does

Documented in `CLAUDE.md` §"Build Division Principle" + the
`restore_after_sync.sh` "一键恢复规则" (single-command recovery rule).

**The build split** (verbatim from `CLAUDE.md` line 254):

> AOSP Soong builds: (a) Java/APK artifacts (framework.jar, etc.),
> (b) host-side tools (dex2oat, aapt2), (c) cross-compiled AOSP
> native .so for OH deployment (libart, libandroid_runtime, libhwui,
> libbase, libcutils, libutils, etc.) → `out/aosp_lib/`.
> OH BUILD.gn builds: adapter's own native artifacts
> (liboh_adapter_bridge.so, libapk_installer.so, appspawn-x) and
> patched OH system services (abilityms, libappms, scene_session_manager).

**Disambiguation rule** (`CLAUDE.md` ~line 270):

> "这个二进制运行在哪里？"
> - DAYU200 device, depends on AOSP source tree → AOSP Soong → `out/aosp_lib/`
> - DAYU200 device, depends on OH IPC/inner_api → OH BUILD.gn → `out/adapter/` or `out/oh-service/`
> - ECS Linux x86_64 host (install/build helper) → AOSP Soong → `out/host-tools/`

**Single-command recovery** (`CLAUDE.md` ~line 220):

> 任何工作成果——包括但不限于源代码、`aosp_patches/`、`ohos_patches/`、
> build 脚本、GN args、ninja patches、配置文件、syscall.h.in 修补、
> stub 模块、被替换的 device .so prebuilt、Python 增量补丁脚本等等——都
> 必须保证：在 AOSP / OH 源码经历一次 `repo sync` 把改动全部覆盖之后，
> 能够通过执行一个 sh 命令，迅速恢复到上一次成功编译时的可用状态。

The recovery script `build/restore_after_sync.sh` is required to be:
- **自包含 (self-contained)**: works from a clean `repo sync`
- **幂等 (idempotent)**: re-runnable without breaking
- **可追溯 (traceable)**: every step in `doc/build_patch_log.html`

**End-to-end build order** (per `CLAUDE.md`):

```
1. repo sync ~/aosp + ~/oh
2. bash build/restore_after_sync.sh        # apply patches, stub modules, GN args
3. cd ~/aosp && bash auto_build.sh         # AOSP Soong: Java jars + .so + dex2oat + boot image
4. cd ~/oh && ./build.sh --product-name rk3568 ... \
       --build-target abilityms libwms libbms ... oh_adapter_bridge appspawn-x
5. bash build/cross_compile_arm32.sh       # glue: libhwui adaptation (40+ scripts)
6. bash build/pull_ecs_artifacts.sh        # ECS → local
7. bash deploy/deploy_to_dayu200.sh        # push 94 files + 4 symlinks
8. reboot + verify pidof foundation render_service
9. hdc file send HelloWorld.apk + bm install + aa start
```

### Why it works

- **One owner per binary** kills the "which build produced this?"
  ambiguity. Disambiguation rule is so explicit you can mechanically
  decide.
- **Single-command recovery** is the **most important engineering
  invariant in the whole project** — see CLAUDE.md *"零孤儿改动"*
  ("zero orphan changes"): any change not encoded in
  `restore_after_sync.sh` doesn't exist. Forces every workaround to
  be a `.patch` or a script step.
- **Append-only `doc/build_patch_log.html`** keeps the rationale
  trail.

### How it adapts to Westlake

We have `prepare_images.sh` (11 steps) and `qemu_boot.sh`, but no
single `restore_after_sync.sh` equivalent. The 一键恢复规则
(single-command recovery) is *the* practice we should borrow before
anything else. Cost is mostly auditing what we currently have.

### Cost to borrow

- **2 days** to author `westlake-restore.sh` covering the patches we
  apply to upstream OHOS, KitKat patches, framework_duplicates rebuilds.
- **1 day** to write the build_patch_log.html equivalent (could be
  Markdown).

### Risk if skipped

We've already paid this cost — handoff_2026-05-12.md vs
handoff_2026-05-14.md show the deltas, but reconstructing the *full*
state from scratch on a new dev box would take a day of detective work.
HBC's rule fixes that to seconds.
